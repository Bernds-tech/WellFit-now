============================================================
0.0 VERBINDLICHER ARBEITSMODUS / ÜBERGABE-PROMPT
============================================================

Diese Datei ist die zentrale Wahrheit für die laufende WellFit-Entwicklung.
Jeder Programmierer, jede Agentur und jeder KI-gestützte Coding-Assistent muss diese Datei zuerst lesen, bevor neue Aufgaben umgesetzt werden.

Arbeitsregel:
Jede neue Idee, jede umgesetzte Karte, jede technische Änderung, jede offene Aufgabe und jede bewusst verschobene Funktion muss in dieser To-do-Liste dokumentiert werden.
So kann später sauber nachgeprüft werden:
- was bereits fertig ist,
- was gerade in Arbeit ist,
- was noch offen ist,
- was bewusst später umgesetzt wird,
- welche Architekturentscheidungen bereits getroffen wurden,
- welche Ideen aufgenommen, aber noch nicht umgesetzt wurden.

Vorgehensweise beim Weiterbauen:
1. Vor jeder neuen Umsetzung diese Datei prüfen.
2. Prüfen, ob die geplante Aufgabe bereits existiert.
3. Wenn ja: Status, Ziel, Umsetzung und Akzeptanzkriterien aktualisieren.
4. Wenn nein: neuen Task mit eindeutiger TASK-ID ergänzen.
5. Danach erst Code ändern.
6. Nach der Umsetzung den Task auf [x] Fertig setzen oder als [~] In Arbeit markieren.
7. Wenn eine Idee entsteht, aber noch nicht gebaut wird, als offenen Task oder Notiz ergänzen.
8. Wenn eine Funktion fachlich verschoben wird, nicht löschen, sondern als späteren Task dokumentieren.
9. Keine produktrelevanten Funktionen nur im Chat besprechen, ohne sie hier festzuhalten.
10. Deployments können gesammelt werden: mehrere Commits sind möglich, danach erst git pull / build / pm2 restart am Server.

Statussystem:
[ ] Offen
[x] Fertig
[~] In Arbeit
[!] Blockiert
[>] Später / Backlog

Server-Deployment erst nach gesammelten Änderungen:
cd /var/www/WellFit-now
git pull origin main
npm run build
pm2 restart wellfit-now

TASK-ID: WF-MOBILE-001
STATUS: [>] Später / Backlog
PRIORITÄT: Grundgerüst / Mobile Roadmap
ZIEL:
Mobile WellFit Companion App als täglichen AR-Begleiter definieren. Die Web-App bleibt die Steuerzentrale, die Mobile App ist für tägliche Bewegung, Kamera/AR, Missionen, Buddy und Fortschritt gedacht.
GRUNDPRINZIP:
Die Mobile App darf keine vollständige Kopie der Web-App werden. Sie zeigt nur die notwendigsten Informationen. Die Startseite ist ein AR-Kamera-Screen, nicht ein klassisches Dashboard.
AR-Elemente dürfen nicht nur flach über das Kamerabild gelegt werden. Sie müssen räumlich in die reale Umgebung eingebettet wirken, so als würden sie sich wirklich dort befinden.
STRATEGIE:
- Zuerst PWA AR-Prototyp bauen.
- Mit echten Usern testen.
- Danach native App prüfen/bauen.
UMSETZUNG:
- Kamera startet nicht automatisch beim Öffnen.
- User klickt zuerst auf Button "AR starten" oder gibt bewusst Kamera-Permission.
- Mobile Startscreen als AR-Kamera-Screen konzipieren.
- Oben links: Hamburger-Menü mit drei Strichen.
- Oben links/mittig neben dem Menü: Buddy-Level anzeigen.
- Oben rechts: kleines Nutzerbild / Profilbild.
- Unten: nur ein Tagesfortschritt-Balken, keine großen Zahlen oder überladene Statistik.
- AR-Objekte räumlich platzieren statt nur als Overlay anzeigen.
- Plane Detection / Flächenerkennung berücksichtigen.
- Anchors / Raumanker berücksichtigen.
- Perspektive, Skalierung und Abstand korrekt darstellen.
- Schatten / Lichtwirkung für digitale Objekte berücksichtigen.
- Objektposition muss beim Bewegen des Smartphones stabil bleiben.
- Später Occlusion prüfen: digitale Objekte können teilweise hinter realen Objekten verschwinden.
- Nur notwendige Informationen anzeigen.
- Web-App bleibt Steuerzentrale, Mobile App bleibt täglicher Begleiter.
AKZEPTANZKRITERIEN:
- Mobile App hat klares eigenes UX-Konzept.
- AR/Kamera steht im Mittelpunkt.
- Kamera startet erst nach bewusster User-Aktion/Permission.
- User sieht Buddy-Level, Profilbild und unten nur den Tagesfortschritt-Balken.
- Digitale Objekte wirken in der realen Umgebung platziert.
- Objekte bleiben im Raum stabil.
- Mobile App bleibt schlank und auf tägliche Nutzung fokussiert.
TECHNISCHE HINWEISE:
- Für PWA/Web-Prototyp WebXR/Three.js prüfen.
- Für Pose Tracking im PWA-Prototyp MediaPipe / TensorFlow.js prüfen.
- Für echte native AR später ARKit/ARCore prüfen.
- Für Pilot zuerst einfacher AR-Modus möglich, aber Ziel bleibt echte räumliche Einbettung.

TASK-ID: WF-TRACKING-001
STATUS: [>] Später / Backlog
PRIORITÄT: Grundgerüst / Echtes Aktivitäts-Tracking
ZIEL:
WellFit muss echte Schritte und echte Aktivitätsdaten aus zuverlässigen Gerätequellen verwenden. GPS alleine reicht nicht als Schrittzähler.
GRUNDPRINZIP:
Keine Fantasie-Schritte. Schritte, aktive Minuten, Workouts und Smartwatch-Daten müssen aus echten Sensoren oder Health-Plattformen kommen und nur nach Zustimmung verarbeitet werden.
UMSETZUNG:
- Handy-Schrittzähler / Pedometer prüfen.
- Beschleunigungssensor / Motion Sensor prüfen.
- Google Fit / Health Connect für Android prüfen.
- Apple HealthKit für iOS prüfen.
- Smartwatch-Daten prüfen: Apple Watch, Pixel Watch, Samsung Watch, Garmin.
- Mögliche Datenpunkte: Schritte, aktive Minuten, Distanz, Herzfrequenz, Kalorien, Workouttyp, Trainingsdauer.
- Datenquelle sichtbar speichern, z. B. source: phone_pedometer, health_connect, apple_health, smartwatch.
- Nur Ergebnisdaten speichern, keine unnötigen Rohdaten.
- Firestore-Struktur für trackingSessions und dailyActivity vorbereiten.
- PWA-Grenzen dokumentieren; native App für vollständige Health-/Watch-Integration einplanen.
AKZEPTANZKRITERIEN:
- Schritte kommen aus echten Gerätesensoren oder Health APIs.
- User erteilt explizite Zustimmung.
- Datenquelle wird angezeigt.
- Keine manuell erfundenen Schritte.
- Daten können für Punkte, Missionen und Tagesfortschritt genutzt werden.

TASK-ID: WF-AI-POSE-001
STATUS: [>] Später / Backlog
PRIORITÄT: Grundgerüst / Mobile KI / Fitnesskern
ZIEL:
WellFit soll über Kamera und Pose Estimation/Skeleton Tracking erkennen, ob User Übungen wie Liegestütze oder Kniebeugen wirklich durchführen und ob die Bewegung grob korrekt ist.
GRUNDPRINZIP:
Die Kamera wird nur nach bewusster Zustimmung genutzt. Es werden keine Videos dauerhaft gespeichert. Es werden nur Ergebnisdaten wie Übung, gültige Wiederholungen, ungültige Wiederholungen, Qualitätssignal und Confidence gespeichert.
Wichtig: Schlechte Wiederholungen werden nicht gezählt.
UMSETZUNG:
- Pose Estimation / Skeleton Tracking im PWA-Prototyp prüfen.
- TensorFlow.js und MediaPipe als Kandidaten prüfen.
- Erste Übungen: Liegestütze und Kniebeugen.
- Körperpunkte erkennen: Kopf, Schultern, Ellbogen, Handgelenke, Hüfte, Knie, Füße.
- Bewegungsphasen erkennen: Startposition, Abwärtsbewegung, Endpunkt, Aufwärtsbewegung, Wiederholung abgeschlossen.
- Liegestütz zählt nur, wenn Bewegungsradius, Körperhaltung und Phasen stimmen.
- Kniebeuge zählt nur, wenn Tiefe, Knie-/Hüftbewegung und Rückkehr in Standposition stimmen.
- Wiederholungen zählen, aber nur gültige Wiederholungen belohnen.
- Ungültige Wiederholungen separat erfassen und nicht vergüten.
- Grobe Bewegungsqualität prüfen: z. B. Rückenhaltung, Bewegungstiefe, zu schnelle/unsaubere Ausführung.
- Buddy gibt kurzes Feedback, warum eine Wiederholung nicht gezählt wurde.
- Buddy gibt motivierendes Feedback, aber keine medizinische Diagnose.
- Nur Ergebnisse speichern, keine Kamerabilder/Videos.
- On-device Verarbeitung bevorzugen.
AKZEPTANZKRITERIEN:
- User stimmt Kamera/Tracking aktiv zu.
- Liegestütz oder Kniebeuge wird erkannt.
- Gültige Wiederholungen werden gezählt.
- Schlechte Wiederholungen werden nicht gezählt.
- Ungültige Wiederholungen erhalten kurzes Buddy-Feedback.
- Ergebnisse können für Punkte/Missionen genutzt werden.
