0.0 VERBINDLICHER ARBEITSMODUS / ÜBERGABE-PROMPT
============================================================

Diese Datei ist die zentrale Wahrheit für die laufende WellFit-Entwicklung.
Jeder Programmierer, jede Agentur und jeder KI-gestützte Coding-Assistent muss diese Datei zuerst lesen, bevor neue Aufgaben umgesetzt werden.

Arbeitsregel:
Jede neue Idee, jede umgesetzte Karte, jede technische Änderung, jede offene Aufgabe und jede bewusst verschobene Funktion muss in dieser To-do-Liste dokumentiert werden.
So kann später sauber nachgeprüft werden:
- was bereits fertig ist,
- was gerade in Arbeit ist,
- was noch offen ist,
- was bewusst später umgesetzt wird,
- welche Architekturentscheidungen bereits getroffen wurden,
- welche Ideen aufgenommen, aber noch nicht umgesetzt wurden.

Vorgehensweise beim Weiterbauen:
1. Vor jeder neuen Umsetzung diese Datei prüfen.
2. Prüfen, ob die geplante Aufgabe bereits existiert.
3. Wenn ja: Status, Ziel, Umsetzung und Akzeptanzkriterien aktualisieren.
4. Wenn nein: neuen Task mit eindeutiger TASK-ID ergänzen.
5. Danach erst Code ändern.
6. Nach der Umsetzung den Task auf [x] Fertig setzen oder als [~] In Arbeit markieren.
7. Wenn eine Idee entsteht, aber noch nicht gebaut wird, als offenen Task oder Notiz ergänzen.
8. Wenn eine Funktion fachlich verschoben wird, nicht löschen, sondern als späteren Task dokumentieren.
9. Keine produktrelevanten Funktionen nur im Chat besprechen, ohne sie hier festzuhalten.
10. Deployments können gesammelt werden: mehrere Commits sind möglich, danach erst git pull / build / pm2 restart am Server.

Statussystem:
[ ] Offen
[x] Fertig
[~] In Arbeit
[!] Blockiert
[>] Später / Backlog

Entwicklungsstrategie:
- Karte für Karte beziehungsweise Feature für Feature umsetzen.
- Nicht zu viele große Änderungen gleichzeitig machen.
- Firestore als produktiven Speicher verwenden.
- localStorage nur für UI-Zustände verwenden.
- Sensible Gesundheits-, Verfassungs- und KI-Personalisierungsdaten müssen mit Hinweistext versehen werden.
- Keine medizinischen Diagnosen oder Heilversprechen.
- Keine Token-, Rendite- oder Investmentversprechen in Nutzerbereichen.
- Neue Ideen werden aufgenommen, aber erst nach Priorität umgesetzt.
- Jede Änderung muss so dokumentiert sein, dass ein anderer Chat oder Entwickler nahtlos fortfahren kann.

Server-Deployment erst nach gesammelten Änderungen:
cd /var/www/WellFit-now
git pull origin main
npm run build
pm2 restart wellfit-now

0.2 NEU AUFGENOMMENE OFFENE AUFGABEN
============================================================

TASK-ID: WF-NAV-001
STATUS: [~] In Arbeit
PRIORITÄT: 1 – sofort als nächstes umsetzen
ZIEL:
Links zu AGB, Datenschutz, Impressum, FAQ und Hilfe müssen im eingeloggten Zustand korrekt funktionieren. Wenn ein eingeloggter User dort auf Zurück klickt, darf er nicht ausgeloggt werden und nicht ungewollt auf der Login-Seite landen.
UMSETZUNG:
- Routen /agb, /datenschutz, /impressum, /faq und /hilfe prüfen.
- Prüfen, ob diese Seiten Auth-State korrekt respektieren.
- Zurück-Button / Browser-History Verhalten prüfen.
- Wenn nötig: Zurück zur Einstellungen- oder Dashboard-Seite statt Login.
- Alle Legal-Links in Sidebar und Footer prüfen.
- Hilfe-Link darf kein toter Text bleiben, sondern muss verlinkt werden.
AKZEPTANZKRITERIEN:
- Eingeloggter User bleibt eingeloggt.
- Zurück führt zurück zur vorherigen App-Seite oder sauber zur App zurück.
- Kein ungewollter Logout.
- Kein Redirect auf Login, wenn Auth-State noch aktiv ist.
- AGB, Datenschutz, Impressum, FAQ und Hilfe funktionieren aus dem eingeloggten Bereich.

TASK-ID: WF-AUTH-002
STATUS: [ ] Offen
PRIORITÄT: 2 – direkt nach WF-NAV-001
ZIEL:
Bereits während der Registrierung soll beim Eingeben des Passworts live angezeigt werden, ob das Passwort stark genug und sicher genug ist.
UMSETZUNG:
- Registrierungsseite prüfen.
- Passwortfeld erweitern.
- Live-Check für Mindestlänge, Groß-/Kleinbuchstaben, Zahlen, Sonderzeichen und Sicherheitsstärke.
- Visuelles Feedback anzeigen, z. B. schwach / mittel / stark.
- Registrierung erst erlauben, wenn Sicherheitsminimum erfüllt ist.
AKZEPTANZKRITERIEN:
- User sieht beim Tippen die Passwortstärke.
- Schwaches Passwort wird erklärt.
- Starkes Passwort wird bestätigt.
- Registrierung verhindert unsichere Passwörter oder warnt mindestens klar.

TASK-ID: WF-REGISTER-002
STATUS: [ ] Offen
PRIORITÄT: 3 – direkt nach WF-AUTH-002
ZIEL:
Alle Informationen, die während der Registrierung abgefragt werden, müssen exakt zur Struktur der Einstellungen passen und nach erfolgreicher Registrierung automatisch in den entsprechenden Einstellungs-Karten sichtbar sein.
UMSETZUNG:
- Registrierungsformular und Firestore-Schreiblogik prüfen.
- Registrierungsfelder mit Settings-Struktur abgleichen.
- Registrierung darf keine abweichenden oder doppelten Feldnamen erzeugen, wenn die Daten später in Einstellungen benötigt werden.
- Daten müssen in diese Zielstruktur geschrieben werden:
  profile: Profilbasis, Biometrie, Vitalwerte, Lifestyle, Aktivität, KI-Buddy
  settings: Benachrichtigungen, Privatsphäre, App-Berechtigungen
- Bereits bestehende Fallback-Kompatibilität aus den Settings berücksichtigen.
- Wenn Registrierungsdaten fehlen, sinnvolle Defaults setzen.
AKZEPTANZKRITERIEN:
- Nach Registrierung sind alle eingegebenen Daten direkt in den Einstellungen sichtbar.
- Änderungen in Einstellungen überschreiben dieselbe Struktur und keine zweite Parallelstruktur.
- Kein erneutes Eingeben derselben Daten nötig.
- Firestore-Userdokument ist sauber strukturiert und konsistent.

STATUS: [ ] Offen
PRIORITÄT: 4 – direkt nach WF-REGISTER-002

UMSETZUNG:

- Inhalte ergänzen: Erste Schritte, Registrierung, Punkte, Missionen, KI-Buddy, Datenschutz, Support.

AKZEPTANZKRITERIEN:

- Eingeloggter User bleibt eingeloggt.
