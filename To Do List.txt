0.0 VERBINDLICHER ARBEITSMODUS / ÜBERGABE-PROMPT
============================================================

Diese Datei ist die zentrale Wahrheit für die laufende WellFit-Entwicklung.
Jeder Programmierer, jede Agentur und jeder KI-gestützte Coding-Assistent muss diese Datei zuerst lesen, bevor neue Aufgaben umgesetzt werden.

Arbeitsregel:
Jede neue Idee, jede umgesetzte Karte, jede technische Änderung, jede offene Aufgabe und jede bewusst verschobene Funktion muss in dieser To-do-Liste dokumentiert werden.
So kann später sauber nachgeprüft werden:
- was bereits fertig ist,
- was gerade in Arbeit ist,
- was noch offen ist,
- was bewusst später umgesetzt wird,
- welche Architekturentscheidungen bereits getroffen wurden,
- welche Ideen aufgenommen, aber noch nicht umgesetzt wurden.

Vorgehensweise beim Weiterbauen:
1. Vor jeder neuen Umsetzung diese Datei prüfen.
2. Prüfen, ob die geplante Aufgabe bereits existiert.
3. Wenn ja: Status, Ziel, Umsetzung und Akzeptanzkriterien aktualisieren.
4. Wenn nein: neuen Task mit eindeutiger TASK-ID ergänzen.
5. Danach erst Code ändern.
6. Nach der Umsetzung den Task auf [x] Fertig setzen oder als [~] In Arbeit markieren.
7. Wenn eine Idee entsteht, aber noch nicht gebaut wird, als offenen Task oder Notiz ergänzen.
8. Wenn eine Funktion fachlich verschoben wird, nicht löschen, sondern als späteren Task dokumentieren.
9. Keine produktrelevanten Funktionen nur im Chat besprechen, ohne sie hier festzuhalten.
10. Deployments können gesammelt werden: mehrere Commits sind möglich, danach erst git pull / build / pm2 restart am Server.

Statussystem:
[ ] Offen
[x] Fertig
[~] In Arbeit
[!] Blockiert
[>] Später / Backlog

Entwicklungsstrategie:
- Karte für Karte beziehungsweise Feature für Feature umsetzen.
- Nicht zu viele große Änderungen gleichzeitig machen.
- Firestore als produktiven Speicher verwenden.
- localStorage nur für UI-Zustände verwenden.
- Sensible Gesundheits-, Verfassungs- und KI-Personalisierungsdaten müssen mit Hinweistext versehen werden.
- Keine medizinischen Diagnosen oder Heilversprechen.
- Keine Token-, Rendite- oder Investmentversprechen in Nutzerbereichen.
- Neue Ideen werden aufgenommen, aber erst nach Priorität umgesetzt.
- Jede Änderung muss so dokumentiert sein, dass ein anderer Chat oder Entwickler nahtlos fortfahren kann.

Server-Deployment erst nach gesammelten Änderungen:
cd /var/www/WellFit-now
git pull origin main
npm run build
pm2 restart wellfit-now

0. ZIEL DIESER DATEI
============================================================

Diese Datei ist die zentrale Arbeitsliste für die laufende WellFit-Entwicklung.
Sie ist nicht als Clickdummy-Plan gedacht.
WellFit befindet sich bereits in der Umsetzung.

Diese Datei soll so genau sein, dass ein externer Programmierer, eine Agentur oder ein KI-gestützter Coding-Assistent direkt erkennen kann:
- was bereits vorhanden ist,
- was noch gebaut werden muss,
- in welcher Reihenfolge weiterprogrammiert werden soll,
- wann eine Aufgabe als fertig gilt,
- welche Datenmodelle, Funktionen und Screens zu berücksichtigen sind,
- welche Punkte später ergänzt werden sollen,
- welche Punkte aktuell bewusst nicht umgesetzt werden.

Diese Datei verbindet:
- das gesamte WellFit-Konzept,
- den aktuellen Produktumsetzungsstand,
- die Firebase-/Firestore-Entscheidung,
- die spätere Solana-/Token-Roadmap,
- Pilotstrategie,
- Businessplan-Logik,
- Investorentranchierung,
- Produkt-, UX-, Marketing-, Partner- und Compliance-Themen,
- konkrete technische Aufgaben für Entwickler und KI-Coding-Tools.

Wichtig:
WellFit soll jetzt zu einer funktionsfähigen, testbaren und pilotfähigen Produktversion weiterentwickelt werden.
Der Fokus liegt nicht mehr auf einem Clickdummy.
Der Fokus liegt auf echter Funktionalität:
- echte User,
- echte Registrierung,
- echte Profile,
- echte Missionen,
- echte Punkte,
- echte History,
- echte Favoriten,
- echte Tracking-Sessions,
- echte Datenhaltung in Firebase/Firestore.

Der Token ist eine spätere Erweiterung und nicht die Voraussetzung für die erste produktiv testbare Version.

Kernlogik:
1. Jetzt Produktfunktionalität fertigstellen.
2. localStorage als Hauptspeicher ablösen.
3. Firebase Auth und Firestore als Startarchitektur nutzen.
4. Pilotfähige Version online bringen.
5. Echte Nutzungsdaten und Feedback sammeln.
6. Danach Partner, LOIs und Investorennachweise weiter ausbauen.
7. Danach Solana, Token, AR, KI und Marketplace sauber erweitern.

0.1 ARBEITSREGELN FÜR PROGRAMMIERER UND KI-CODING-ASSISTENTEN
============================================================

Jede Aufgabe muss nach diesem Muster bearbeitet werden:

TASK-ID:
Kurzer eindeutiger Name, z. B. WF-AUTH-001.

ZIEL:
Was soll nach der Umsetzung funktionieren?

UMSETZUNG:
Welche Dateien, Komponenten, Collections, Funktionen oder Screens sind betroffen?

AKZEPTANZKRITERIEN:
Wann gilt die Aufgabe als fertig?

STATUS:
[ ] Offen
[x] Fertig
[~] In Arbeit
[!] Blockiert

REGELN:
[ ] Keine neue Logik mehr nur in localStorage speichern, wenn sie produktrelevant ist.
[ ] Userdaten, Punkte, Missionstatus, Favoriten, History und Tracking immer in Firebase/Firestore speichern.
[ ] localStorage nur für UI-Zustände verwenden, z. B. Theme, Sidebar offen/geschlossen, temporäre Filter.
[ ] Keine Secrets, API Keys oder private Tokens sichtbar ins Repository committen.
[ ] Jede neue Funktion muss mobil und desktopfähig sein.
[ ] Jede neue Funktion muss mindestens manuell getestet werden.
[ ] Jede neue Funktion braucht klare Fehlermeldungen.
[ ] Jede neue Funktion muss so geschrieben werden, dass sie später leichter in eine App/PWA überführt werden kann.
[ ] Keine Token-/Investment-Versprechen in Nutzerbereichen.
[ ] Keine medizinischen Diagnosen oder Heilversprechen.

Definition of Done für technische Tasks:
[ ] Feature ist implementiert.
[ ] Feature ist im UI erreichbar.
[ ] Daten werden korrekt gespeichert oder gelesen.
[ ] Fehlerfälle sind behandelt.
[ ] Mobile Ansicht wurde geprüft.
[ ] Desktop Ansicht wurde geprüft.
[ ] Keine Konsolenfehler.
[ ] Keine offensichtlichen toten Links.
[ ] Code ist verständlich benannt.
[ ] Aufgabe wurde in dieser Liste auf [x] gesetzt oder als erledigt dokumentiert.
