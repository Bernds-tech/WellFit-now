0. ZIEL DIESER DATEI
============================================================

Diese Datei ist die zentrale Arbeitsliste für die laufende WellFit-Entwicklung.
Sie ist nicht als Clickdummy-Plan gedacht.
WellFit befindet sich bereits in der Umsetzung.

Diese Datei soll so genau sein, dass ein externer Programmierer, eine Agentur oder ein KI-gestützter Coding-Assistent direkt erkennen kann:
- was bereits vorhanden ist,
- was noch gebaut werden muss,
- in welcher Reihenfolge weiterprogrammiert werden soll,
- wann eine Aufgabe als fertig gilt,
- welche Datenmodelle, Funktionen und Screens zu berücksichtigen sind,
- welche Punkte später ergänzt werden sollen,
- welche Punkte aktuell bewusst nicht umgesetzt werden.

Diese Datei verbindet:
- das gesamte WellFit-Konzept,
- den aktuellen Produktumsetzungsstand,
- die Firebase-/Firestore-Entscheidung,
- die spätere Solana-/Token-Roadmap,
- Pilotstrategie,
- Businessplan-Logik,
- Investorentranchierung,
- Produkt-, UX-, Marketing-, Partner- und Compliance-Themen,
- konkrete technische Aufgaben für Entwickler und KI-Coding-Tools.

Wichtig:
WellFit soll jetzt zu einer funktionsfähigen, testbaren und pilotfähigen Produktversion weiterentwickelt werden.
Der Fokus liegt nicht mehr auf einem Clickdummy.
Der Fokus liegt auf echter Funktionalität:
- echte User,
- echte Registrierung,
- echte Profile,
- echte Missionen,
- echte Punkte,
- echte History,
- echte Favoriten,
- echte Tracking-Sessions,
- echte Datenhaltung in Firebase/Firestore.

Der Token ist eine spätere Erweiterung und nicht die Voraussetzung für die erste produktiv testbare Version.

Kernlogik:
1. Jetzt Produktfunktionalität fertigstellen.
2. localStorage als Hauptspeicher ablösen.
3. Firebase Auth und Firestore als Startarchitektur nutzen.
4. Pilotfähige Version online bringen.
5. Echte Nutzungsdaten und Feedback sammeln.
6. Danach Partner, LOIs und Investorennachweise weiter ausbauen.
7. Danach Solana, Token, AR, KI und Marketplace sauber erweitern.

1. STRATEGISCHE GRUNDENTSCHEIDUNG
============================================================

[ ] WellFit nicht als reine Krypto-App positionieren.
[ ] WellFit als Health-, Bewegung-, Lern-, Gamification- und Engagement-Plattform positionieren.
[ ] Token erst als spätere Ökosystem- und Belohnungskomponente darstellen.
[ ] Erste Version mit Punkten, Leveln, Missionen, Badges, History, Favoriten und Rewards weiterentwickeln.
[ ] Blockchain/Token nur dort erwähnen, wo es strategisch sinnvoll und rechtlich sauber ist.
[ ] Alle alten SUI-Verweise prüfen und auf Solana korrigieren, weil WellFit Solana verwenden soll.
[ ] Keine Renditeversprechen verwenden.
[ ] Keine medizinischen Heilversprechen verwenden.
[ ] Keine Aussagen machen, die WellFit unnötig in Richtung Finanzprodukt oder Medizinprodukt schieben.

Kurzpositionierung:
WellFit ist eine gamifizierte Gesundheits- und Bewegungsplattform, die reale Aktivität, Aufgaben, Lernen, soziale Interaktion und digitale Belohnungen verbindet. In der ersten funktionsfähigen Version sammeln Nutzer Punkte, Level und Status durch Bewegung und Challenges. In späteren Phasen werden diese Mechaniken mit Token, AR-Erlebnissen, KI-Buddy, Partnerangeboten und einem digitalen Marktplatz erweitert.

4. FIREBASE / FIRESTORE ENTSCHEIDUNG
============================================================

Für Phase 1 sehr sinnvoll.

Wenn das Ziel jetzt ist:
- echte User statt localStorage,
- echte History,
- echte Favoriten,
- erste produktive Version,
- Tracking-Daten sicher speichern,
- schnell online gehen,

dann ist Firebase absolut sinnvoll.

Für Phase 2/3 vielleicht nicht ideal, wenn das Ziel später ist:
- skalierbare Game Engine,
- komplexe Rewards,
- Anti-Cheat,
- KI-gesteuerte Missionserzeugung,
- starke Datenanalyse,
- hohe Regelkontrolle,
- viele Tracking-Events.

Plan:
[ ] Firebase Auth + Firestore jetzt verwenden.
[ ] echte Registrierung/Login umsetzen.
[ ] User-Profile in Firebase speichern.
[ ] Missionen in Firestore speichern.
[ ] Favoriten in Firestore speichern.
[ ] History in Firestore speichern.
[ ] Tracking-Sessions in Firestore speichern.
[ ] GPS-/Event-Daten zunächst schlank speichern.
[ ] Server in Singapur nur für die App verwenden, falls bereits so geplant.
[ ] Next.js dort deployen, falls Next.js verwendet wird.
[ ] keine eigene DB-Administration am Anfang.
[ ] weniger DevOps, weniger Fehlerquellen.
[ ] Regel: keine Game-Logik mehr im localStorage.
[ ] localStorage nur noch für UI-Dinge verwenden.
[ ] Punkte, Rewards, Missionstatus und History immer in Firebase speichern.
[ ] Firebase-Datenmodell bewusst einfach halten.
[ ] nicht jeden GPS-Punkt endlos speichern.
[ ] Sessions und aggregierte Events speichern.
[ ] Reads/Writes im Blick behalten, weil Firestore nach Operationen abrechnet.
[ ] Später migrieren, wenn nötig.
[ ] Wenn WellFit groß wird, Teile später auf eigenes Backend + SQL ziehen.

Merksatz:
Firebase ist ein gutes Startproblem-Lösungsmittel. Skalierung ist ein späteres Problem.
