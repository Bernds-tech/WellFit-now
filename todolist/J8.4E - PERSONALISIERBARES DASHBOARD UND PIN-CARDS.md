# J8.4E – Personalisierbares Dashboard und Pin-Cards

Stand: 2026-05-05

## Produktidee

Der Nutzer soll sein Dashboard selbst zusammenstellen können.

Nicht jede Karte und nicht jedes Modul muss dauerhaft auf dem Dashboard sichtbar sein. Der Nutzer entscheidet selbst, welche Karten wichtig sind. Alle anderen Funktionen bleiben weiterhin über Sidebar und Unterseiten erreichbar.

## Kernmechanik: Pin zum Dashboard

[x] Jede geeignete Karte auf der gesamten Website soll später ein kleines Auswahl-Element erhalten.
[x] Dieses Element ist visuell als Kreis mit Punkt in der Mitte gedacht.
[x] Funktional entspricht es einem „Zum Dashboard hinzufügen“-Pin.
[x] Vergleichbar mit Favoriten/Stern, aber Ziel ist nicht Favoritenliste, sondern Dashboard-Sichtbarkeit.

Beispiel:

```txt
[●] Buddy-Statuskarte      -> wird im Dashboard angezeigt
[○] Marktplatzkarte        -> wird nicht im Dashboard angezeigt
[●] Bedürfnisse / Needs    -> wird im Dashboard angezeigt
[○] Leaderboard            -> bleibt nur über Sidebar erreichbar
```

## Dashboard-Verhalten

[x] Dashboard wird zur persönlichen Startzentrale.
[x] Nutzer kann Buddy anzeigen lassen oder ausblenden.
[x] Nutzer kann Bedürfnisse / Needs anzeigen lassen oder ausblenden.
[x] Nutzer kann Missionen, Punkte, Analytics, Leaderboard, Marktplatz, Punkte-Shop usw. auswählen.
[x] Nicht ausgewählte Module verschwinden nur vom Dashboard, bleiben aber über ihre eigenen Seiten erreichbar.

## Skalierbare Kartenregel

[!] Jede Dashboard-Karte muss skalierbar bleiben.
[!] Karten dürfen in verschiedenen Größen dargestellt werden können.
[!] Inhalte dürfen bei Größenänderung nicht einfach verschwinden.
[!] Stattdessen müssen sie responsiv umfließen, komprimieren oder in klaren Unterbereichen angezeigt werden.
[!] Wichtige Daten bleiben immer sichtbar.

Geplante Größen:

```txt
small   = kompakte Übersicht
medium  = Standard-Karte
large   = Detailkarte
wide    = breite Fokuskarte
```

## Datenmodell-Vorschlag

Speicherort im User-Dokument:

```ts
settings: {
  dashboard: {
    pinnedCards: [
      "buddyStatus",
      "userNeeds",
      "dailyMission",
      "pointsBalance"
    ],
    cardSizes: {
      buddyStatus: "large",
      userNeeds: "medium",
      dailyMission: "wide",
      pointsBalance: "small"
    },
    cardOrder: [
      "dailyMission",
      "buddyStatus",
      "userNeeds",
      "pointsBalance"
    ]
  }
}
```

```ts
id

description
category

requiresAuth

```

## Pin-Komponente

[ ] Gemeinsame Pin-Komponente planen:

```txt
app/components/DashboardPinToggle.tsx
```

Aufgabe:

```txt
- zeigt Kreis mit Punkt
- aktiv/inaktiv Zustand
- toggelt Karte ins Dashboard
- schreibt später in settings.dashboard.pinnedCards
- kein Reward-/Punkte-/Token-Effekt
```

## Wichtige Sicherheitsregel

[!] Pinning ist nur eine UI-/Preference-Funktion.
[!] Pinning darf niemals Punkte, XP, Rewards, Wallet, Token, NFT, Shop-Käufe oder Leaderboard-Wertungen autorisieren.
[!] Daten, die kritisch sind, bleiben serverseitig validiert.

## Offene Umsetzungsschritte

[ ] Vor Umsetzung vorhandene Dashboard-Komponenten vollständig prüfen.
[ ] Dashboard Card Registry anlegen.
[ ] DashboardPinToggle als reine UI-Komponente anlegen.
[ ] Dashboard Preferences zunächst lokal oder Firestore-safe vorbereiten.
[ ] Später Firestore-Anbindung unter `settings.dashboard` mit onSnapshot.
[ ] Dashboard-Karten responsive Größenlogik definieren.
[ ] Erste Karten: BuddyStatus, UserNeeds, DailyMission, PointsBalance, Analytics, Marketplace, Leaderboard, PointsShop.
[ ] Build nach jeder kleinen Änderung ausführen.
